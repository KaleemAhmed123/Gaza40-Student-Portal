## Findings (ranked, biggest cut first)

### 1. Duplicate file-upload path in storage layer
- **What:** `uploadFileToStorage()` in [src/shared/storage.ts:52](src/shared/storage.ts#L52) re-implements the exact same key-generation and local-disk-fallback logic as `uploadToStorage()` — just to accept a file path instead of an in-memory `Buffer`.
- **Only caller:** `uploadCsvFile()` in [src/modules/csv-generator/csv-storage.service.ts:29](src/modules/csv-generator/csv-storage.service.ts#L29), itself only called once, from `csv-job.service.ts:182`.
- **Risk if left alone:** any future change to the key format or local-fallback behavior (e.g. adding a hash, changing the folder structure) has to be made in two places or they silently drift apart.
- **Fix:** at the one call site, read the CSV file into a `Buffer` (`fs.readFile`) and call the existing `uploadToStorage` / `uploadCsv`. Delete `uploadFileToStorage` and `uploadCsvFile`.
- **Savings:** ~35 duplicate lines across 2 files, 0 dependency change.

### 2. Three role-gate middleware functions doing the same check
- **What:** `requireRole()`, `requireAnyActiveDbRole()`, and `requireActiveDbRole()` in [src/middleware/auth.middleware.ts:38-69](src/middleware/auth.middleware.ts#L38-L69) all run the identical logic — check `req.authUser.roles` against an allowed list — under three different names. None of them actually queries the database despite the "DbRole" naming.
- **Spread:** 15 call sites split across the three (`requireRole` ×10, `requireActiveDbRole` ×3, `requireAnyActiveDbRole` ×2).
- **Risk if left alone:** a future fix (e.g. also checking `accountStatus`) has to be found and applied in three places; easy to pick the "wrong" one since none of them behave differently today.
- **Fix:** keep one function (e.g. `requireRole(roles: RoleCode[])`), update the 15 call sites, delete the other two.
- **Note:** `requireActiveMentor` (9 call sites) is a legitimate, well-used shorthand for the common mentor-only case — not flagged.
