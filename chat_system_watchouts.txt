Top Real-World Chat Bugs
1. Permission Checks Only On Frontend

Very common.

Bad:

Hide button if mentor

Good:

Frontend hides button
Backend verifies permission again

Never trust frontend permissions.

A Mentor should never be able to call an API directly and bypass UI restrictions.

2. Socket Authentication Missing

Bad:

User connects socket
Socket trusted

Good:

JWT validated during socket connection
User identity attached to socket

Every socket connection must be authenticated.

3. Duplicate Conversations

Example:

Admin → Mentor
Mentor → Admin

Creates:

Conversation A
Conversation B

for same participants.

Always check whether a direct conversation already exists.

4. Duplicate Messages

Very common.

User sends message.

Network lag.

Client retries.

Message saved twice.

Solution:

clientMessageId
idempotency checks
5. Unauthorized Group Access

Example:

Regional Admin manually sends API request:

Add Mentor from another region

Frontend blocks it.

API doesn't.

Result:

Permission bypass.

All role rules must live in backend.

6. Missing Membership Validation

Example:

User sends:

{
  "conversationId": "123"
}

Server accepts.

Never verifies user belongs to conversation.

Result:

People can send/read messages from groups they are not members of.

Critical bug.

7. Orphaned Messages

Group deleted.

Messages remain.

Attachments remain.

Storage keeps growing.

Need cleanup strategy.

8. Attachment Abuse

Users upload:

100 MB ZIP
.exe file
malware

Need:

Size limits
Mime validation
Whitelist extensions
Virus scanning if needed
9. Message Ordering Issues

Common with Socket.IO.

Messages appear:

Message 2
Message 1
Message 3

Always sort by:

createdAt
server timestamp

Never trust client timestamp.

10. Missing Reconnection Handling

User loses internet.

Reconnects.

Messages missing.

Need:

Fetch missed messages

after reconnect.

# Chat System Engineering Rules

The chat system must prioritize correctness, security, scalability, and simplicity over feature richness.

Mandatory requirements:

* Never trust frontend permissions. Every permission must be enforced on the backend.
* All Socket.IO connections must be authenticated using JWT.
* A user must never be able to read, send, edit, or delete messages from conversations they do not belong to.
* Prevent duplicate direct conversations between the same participants.
* Prevent duplicate messages caused by retries, reconnects, or network failures.
* All role validations must happen on the server.
* Regional Admins must never be able to add mentors from other regions unless explicitly allowed by business rules.
* Mentors must never be able to communicate with other mentors unless requirements change.
* All message timestamps must be generated by the server.
* Message ordering must be based on server timestamps, not client timestamps.
* File uploads must validate size, MIME type, and allowed extensions.
* Messages and attachments must support retention and archival policies.
* Socket reconnection must be handled gracefully without message loss.
* All database queries used in chat must be indexed appropriately.
* Chat architecture must be designed to support future Student roles without major refactoring.
* Authorization, membership validation, and role validation must be treated as critical security requirements.
* Simplicity is preferred over advanced features. Do not introduce unnecessary complexity, abstractions, or microservices unless justified by scale.
